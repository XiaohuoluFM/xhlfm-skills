# 中文播客雷达 Skill 设计文档

- Status: Implemented v0
- Updated: 2026-04-01
- Skill id: `podcast-radar-cn`

## 1. 设计目标

把《中文播客榜》的榜单数据转换为适合真实工作流使用的结果。

Skill 不应该只返回“榜上前 N 名”，而应该根据用户意图，输出更适合决策和行动的内容：

- 听众想知道听什么
- 创作者想知道研究谁
- 自媒体想知道推荐谁
- 二创用户想知道拆哪一期

## 2. 设计原则

### 2.1 榜单字段与标题优先

第一版默认优先使用下面这些低成本信号完成大部分判断：

- 榜单类型
- 排名
- 标题
- 播客名
- 分类
- 播放/评论/订阅等基础热度字段
- 更新时间相关字段

也就是说，能从标题和基础字段里推断出来的信息，就不依赖详情页。

### 2.2 详情富化是少量按需步骤

详情页富化只用于补充必要上下文，例如：

- 节目简介不够明确
- 需要从单集页推导真实小宇宙 `pid`
- 用户明确要求对少量候选结果做更细的推荐或分发判断

### 2.3 输出要面向任务，不面向字段

不要把 API 字段原样堆给用户，而是转成：

- 推荐理由
- 可点击入口
- 筛选后的清单
- 创作或分发场景下的结论

### 2.4 中文口语优先解释

在中文语境里，用户说“热门播客”时，经常真正想要的是“最近哪几期最值得点开”。

因此第一版采用下面这条解释规则：

- 默认把泛化的“热门播客”“最近播客”“推荐几个播客”理解为节目 / 单集级发现请求
- 只有在用户明确提到“播客频道”“播客栏目”“节目主页”“频道级榜单”时，才切到播客频道级结果
- 创作者对标、频道观察、栏目研究这类任务，也优先使用播客级结果

### 2.5 尊重数据边界

这个 Skill 解决的是“最新一轮榜单里的热门/新锐发现”，不是全平台、实时、绝对客观的热度判断。

### 2.6 严格保护目标站点

Skill 设计必须默认避免对小宇宙页面做瞬时或短时间批量访问。

约束原则：

- 默认不批量抓详情页
- 默认不为了“补全所有字段”而访问详情页
- 只有在候选结果已经收敛到少量对象时，才允许按需访问详情页
- 对需要大规模详情抓取的请求要严格拒绝或降级为榜单级结果

## 3. 数据源设计

## 3.1 榜单源

| 类型 | 接口 | 主要对象 |
| --- | --- | --- |
| 热门节目 | `https://xyzrank.com/api/episodes` | 单集 |
| 热门播客 | `https://xyzrank.com/api/podcasts` | 播客 |
| 新锐节目 | `https://xyzrank.com/api/new-episodes` | 单集 |
| 新锐播客 | `https://xyzrank.com/api/new-podcasts` | 播客 |

所有接口都支持：

- `offset`
- `limit`

返回结构统一为：

```json
{
  "items": [],
  "total": 1000,
  "limit": 50,
  "offset": 0
}
```

## 4. 关键链路设计

## 4.1 从热门单集推导播客详情

榜单单集接口只提供：

- `podcastID`
- 小宇宙单集链接

但不直接提供小宇宙播客页 URL。

更可靠的链路是：

1. 先取单集链接
2. 拉取小宇宙单集页
3. 从 `episode.pid` 得到播客真实 `pid`
4. 拼出播客页 URL：`https://www.xiaoyuzhoufm.com/podcast/<pid>`

说明：

- 不能假设榜单里的 `podcastID` 就是小宇宙 `pid`
- 榜单里的 `podcastID` 与播客榜接口里的 `id` 是对齐的，但与小宇宙 `pid` 不是同一个字段体系

## 4.2 推荐结果生成链路

```text
用户需求
  -> 先做中文意图映射（默认“播客” -> 节目 / 单集）
  -> 选择榜单类型
  -> 拉取榜单数据
  -> 按分类/新鲜度/数量做基础筛选
  -> 优先基于标题和基础字段生成候选判断
  -> 只有在必要时，对少量候选对象做详情页富化
  -> 生成面向场景的输出
```

## 4.3 标题优先提取策略

第一版优先从标题中提取这些信息：

- 节目期号：如 `Vol`、`EP`、`S8E9`
- 嘉宾线索：如 `A×B`、`对话某某`、`with 某某`
- 主题线索：如商业、科技、职场、情感、历史等显性词
- 形式线索：如对谈、独白、访谈、盘点、复盘

这一步的目标不是做完美语义分析，而是尽量减少对外部详情页的依赖。

## 4.3A 用户语义映射策略

默认映射：

- `热门播客`
- `最近热门播客`
- `最近值得听的播客`
- `推荐几个播客`

优先落到：

- `hot-episodes`
- `new-episodes`

明确频道级映射：

- `播客频道`
- `播客栏目`
- `节目主页`
- `频道级榜单`
- `栏目级榜单`

优先落到：

- `hot-podcasts`
- `new-podcasts`

创作者研究类例外：

- 即便用户用了“播客”这个词，只要任务明显是在找对标对象、栏目样本、频道研究对象，也优先走播客级结果

## 4.4 详情页访问策略与风控保护

这是实现时必须遵守的硬规则。

### 访问条件

只有同时满足下面条件，才允许访问小宇宙详情页：

1. 榜单基础字段和标题不足以完成当前任务
2. 候选对象已经收敛到少量结果
3. 当前访问是为了补充必要上下文，而不是批量采集

### 单次富化上限

- 单轮任务中，详情页连续访问上限设为 `20`
- 推荐默认远低于这个上限，优先控制在 `5-10`

### 严格拒绝的情况

- 用户要求一次性补全大量榜单对象详情
- 用户要求在短时间内批量抓取几十到上百个详情页
- 用户要求遍历多页榜单并为所有结果抓取详情
- 任何带有明显批量采集特征的详情请求

### 降级策略

当触发上述限制时，Skill 不继续抓详情，而是退回到：

- 仅使用榜单和标题完成推荐
- 缩小结果数量后再做少量富化
- 输出“榜单级发现结果”而不是“详情级研究结果”

## 5. 输出模式设计

## 5.1 听众发现模式

输入示例：

- 最近最热门的中文播客有哪些？
- 给我 10 个最近值得听的商业类中文播客

输出结构建议：

1. 简短结论
2. 推荐清单
3. 每项包含：
   - 单集 / 节目名
   - 类型/分类
   - 热度信号
   - 为什么值得听
   - 收听链接

## 5.2 创作者对标模式

输入示例：

- 我做商业播客，帮我找最近可以对标的热门节目
- 看看最近新锐播客里有哪些值得关注的访谈节目

输出结构建议：

1. 对标对象列表
2. 每项包含：
   - 节目名
   - 主播/机构
   - 分类
   - 更新新鲜度
   - 热门单集样本
   - 可借鉴点

## 5.3 策展与分发线索模式

输入示例：

- 帮我整理一份最近适合推荐给职场人的中文播客
- 帮我挑出最近有传播潜力的访谈类单集

输出结构建议：

1. 节目或单集列表
2. 每项包含：
   - 节目名或单集标题
   - 播客名
   - 热度信号
   - 推荐理由
   - 可策展/可二创角度
   - 原链接

## 6. 数据字段使用策略

## 6.1 可以直接展示的字段

- `rank`
- `title` / `name`
- `primaryGenreName`
- `authorsText`
- `playCount`
- `commentCount`
- `subscription`
- `avgPlayCount`
- `avgCommentCount`
- `lastReleaseDateDayCount`
- 外部链接

## 6.2 需要谨慎解释的字段

### `openRate` / `avgOpenRate`

实测存在大于 `1` 的情况，因此第一版不解释成“打开率”或“完播率”。

建议处理方式：

- 不直接向普通用户展示原值
- 内部仅把它作为一个可选热度参考信号
- 若必须展示，统一改称“平台开放度信号”或“榜单参考信号”

### `primaryGenreName`

在新锐榜中缺失较多。

建议处理方式：

- 为空时标记为 `未分类`
- 需要分类推荐时，优先用已有分类；缺失时可结合详情页描述辅助判断，但第一版不做自动重分类

## 8. 计划中的 Skill 行为

Skill 被触发后，推荐流程如下：

1. 判断用户目标更像听众发现、创作者研究还是策展与分发线索
2. 选择合适的榜单：
   - 热门播客
   - 热门单集
   - 新锐播客
   - 新锐单集
3. 拉取足够数量的候选数据
4. 优先用标题和基础字段完成筛选与推荐
5. 如有必要，且候选结果不超过上限时，再抓取少量详情页补充上下文
6. 输出面向场景的发现结果

## 10. 已知风险与待验证问题

### 10.1 风险

- 榜单周更，不适合被包装为“实时热搜”
- 新锐榜分类缺失较多
- 详情页富化链路比直接 API 更慢
- 平台字段命名与含义并不完全标准化
- 如果不限制富化规模，容易触发目标站点风控

### 10.2 待验证问题

- 二创场景是否需要进一步提取时间戳与片段主题
- 听众模式是否需要加入更明确的“入门推荐”逻辑

## 11. 推荐实现优先级

### Phase 1

- 热门/新锐四类榜单拉取
- 基础筛选
- 标题优先的信息提取
- 少量详情页富化
- 三种输出模式

### Phase 2

- 更好的分类兜底
- 更细的推荐理由模板
- 二创场景下的时间戳提取和摘录结构化
